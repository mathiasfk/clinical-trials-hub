## Why

The MVP backend was implemented in Go to deliver the prototype quickly, but the team has chosen .NET 10 as the long-term server platform for the Clinical Trials Hub. Porting the backend now â€” while the API surface, domain rules, and data shapes are still small and sufficiently captured by the Go-generated OpenAPI artifacts at `backend/docs/swagger.{json,yaml}` (used only as an initial reference) â€” keeps the migration cheap and avoids accumulating Go-specific debt. The new backend keeps the existing front-end working unchanged and prepares the codebase for a future PostgreSQL migration by going through EF Core from day one.

## What Changes

- **BREAKING (developer-facing only)**: replace the Go backend at `backend/` with a new .NET 10 implementation in the same directory. The existing Go sources have already been moved to `backend-go-backup/` and remain only as a behavioral reference.
- Introduce a four-project hexagonal solution under `backend/`:
  - `ClinicalTrialsHub.Domain` (entities, value objects, registry of eligibility dimensions, allow-listed vocabularies â€” no external dependencies).
  - `ClinicalTrialsHub.Application` (use-case services, repository contracts, validation, ID-generation contract).
  - `ClinicalTrialsHub.Infrastructure` (EF Core `DbContext`, EF-Core-backed `IStudyRepository` implementation using `Microsoft.EntityFrameworkCore.InMemory`, sequential `study-NNNN` ID generator, deterministic seed catalog). EF Core types MUST NOT leak outside this project.
  - `ClinicalTrialsHub.Api` (Minimal API host, endpoint groups, JSON/CORS/native-OpenAPI configuration, Scalar API reference UI, exception-handling middleware, health check).
  - Plus `ClinicalTrialsHub.Tests` (NUnit + NSubstitute) covering the domain, application, and HTTP layers.
- Reproduce the HTTP surface exposed by the previous Go backend byte-compatibly from the front-end perspective: same paths, same camelCase JSON, same `{ message, errors }` error envelope, same status codes, same `unknown field` rejection on JSON deserialization, same `study-NNNN` ID format, same seed catalog (one study per therapeutic area, contiguous IDs starting at `study-0001`), same `/health` shape `{ "status": "ok" }`. The front-end typings in `frontend/src/types.ts` are the authoritative contract; the existing `backend/docs/swagger.{json,yaml}` files generated by the Go server serve only as initial reference.
- Describe the API with the native ASP.NET Core OpenAPI stack (`Microsoft.AspNetCore.OpenApi` â€” `builder.Services.AddOpenApi()` + `app.MapOpenApi()`) instead of Swashbuckle. The runtime OpenAPI document is served at `/openapi/v1.json`. Expose an interactive API reference UI via Scalar (`Scalar.AspNetCore` â€” `app.MapScalarApiReference()`) instead of Swagger UI.
- Export the current OpenAPI document to `backend/docs/openapi.json` (and optionally `backend/docs/openapi.yaml`) via the `Microsoft.Extensions.ApiDescription.Server` build-time generator, so downstream tooling (front-end developers, NSwag/OpenAPI client generators) can read the contract from a known file path. If the native .NET OpenAPI output diverges in shape from the Go-generated `backend/docs/swagger.{json,yaml}` files (for example OpenAPI 3.1 vs 3.0), adopt the new shape and delete the legacy reference files; they are not part of the steady-state contract.
- Provide a Dockerfile and `fly.toml` suitable for deploying the new backend to fly.io and a CORS configuration that can be tightened for the production Vercel front-end origin.
- Keep the entire AI-assistant heuristic on the front-end (no new backend endpoints for the assistant). The backend continues to expose only the read-only `GET /api/studies` and `GET /api/studies/:id` calls the assistant relies on.

## Capabilities

### New Capabilities

_None._ This change is a re-implementation of existing behavior on a new stack; it does not introduce a new product capability.

### Modified Capabilities

- `platform-bootstrap`: the backend stack requirement changes from "Go" to ".NET 10 with ASP.NET Core Minimal API"; the persistence-boundary requirement is reaffirmed in the .NET context and tightened to require that the repository implementation go through EF Core (using the in-memory provider for the MVP) with EF Core types confined to the Infrastructure project; the OpenAPI requirement is updated to use the native ASP.NET Core OpenAPI stack with Scalar as the interactive UI; English-language conventions remain unchanged.

## Impact

- **Code**: full replacement of `backend/`. No source files survive from the Go implementation; `backend-go-backup/` is the read-only reference.
- **APIs**: HTTP contract is preserved exactly. No front-end change is required; `frontend/src/api.ts` and `frontend/src/types.ts` continue to compile and work.
- **OpenAPI artifacts**: the .NET project produces a new artifact at `backend/docs/openapi.json` via the build-time OpenAPI generator. The legacy Go-generated `backend/docs/swagger.{json,yaml}` files serve only as initial reference and MAY be deleted once the .NET-generated artifact is in place (especially if the OpenAPI version/shape differs).
- **Dependencies**: introduces .NET 10 SDK, ASP.NET Core, `Microsoft.AspNetCore.OpenApi` (native OpenAPI), `Microsoft.Extensions.ApiDescription.Server` (build-time OpenAPI document generation), `Scalar.AspNetCore` (interactive API reference UI), `Microsoft.EntityFrameworkCore`, `Microsoft.EntityFrameworkCore.InMemory`, `FluentValidation` (or `FluentValidation.AspNetCore`), `NUnit`, `NSubstitute`, `Microsoft.AspNetCore.Mvc.Testing`. Explicitly does NOT use `Swashbuckle.AspNetCore`. Removes Go toolchain from the backend build path.
- **Deployment**: existing Go `Dockerfile` and `fly.toml` (in `backend-go-backup/`) are superseded by the new .NET equivalents under `backend/`. CORS allowed origins are configurable so the production Vercel domain can be added without code changes.
- **Tests**: new NUnit-based test project replaces the Go test files in `backend-go-backup/`. CI will need a .NET 10 SDK image instead of Go.
- **Persistence**: still in-memory, but now implemented via `Microsoft.EntityFrameworkCore.InMemory` behind an EF-Core `DbContext` inside the Infrastructure layer. The repository contract lives in the Application layer and remains agnostic to the storage technology; switching to PostgreSQL is a swap of the EF Core provider (plus migrations) in the Infrastructure layer only, with no domain or application changes.

## Change log

- **`backend-go-backup/`** remains the legacy Go reference implementation. It will be removed in a **follow-up change** after the .NET backend has shipped to production and the team no longer needs side-by-side behavioral comparison.
