## Context

This is a green-field project that must produce a demonstrable MVP within one day (6-8 hours). The MVP must prioritize delivery speed and developer familiarity while preserving a clean path toward a production-grade stack in later phases.

The requested stack is:
- Frontend: React + Vite + pnpm
- Backend: Go
- Persistence: in-memory for MVP, with explicit compatibility for a future Postgres-backed implementation

The first capability slice is "Study registration" from `docs/features-outline.md`, including objectives, endpoints, inclusion/exclusion criteria, participant count, study type, number of arms, phase, therapeutic area, and patient population.

## Goals / Non-Goals

**Goals:**
- Create a project foundation that supports rapid local MVP development.
- Implement a clear frontend-backend contract for study registration.
- Keep all repository text artifacts in English (code, docs, specs, README content).
- Isolate persistence behind interfaces so storage can evolve from in-memory to Postgres with minimal domain refactor.

**Non-Goals:**
- Implement production-ready infrastructure (auth hardening, distributed systems concerns, advanced observability).
- Implement non-registration features from the outline (SOA validation, similar studies detection, insights, AI assistant, decision log).
- Complete data migration scripts to Postgres in this change.

## Decisions

1. **Monorepo structure with separate frontend/backend apps**
   - **Decision:** Organize the repository with clear application boundaries (frontend app and backend app) and shared documentation under the existing repository.
   - **Rationale:** Supports independent iteration speed while keeping onboarding and local development simple.
   - **Alternatives considered:** Single binary serving static frontend assets from Go. Rejected for MVP because it slows UI iteration.

2. **Frontend stack: React + Vite + pnpm**
   - **Decision:** Use React with Vite and pnpm workspace-compatible tooling.
   - **Rationale:** Fast startup/build loops and user familiarity reduce implementation risk under strict time constraints.
   - **Alternatives considered:** Next.js. Rejected because SSR and framework conventions add non-essential setup overhead for this MVP.

3. **Backend stack: Go with layered architecture**
   - **Decision:** Implement HTTP handlers, application services, and repository interfaces as separate layers.
   - **Rationale:** Keeps business rules testable and storage-agnostic.
   - **Alternatives considered:** Minimal single-file handler implementation. Rejected because it creates migration friction for Postgres adoption.

4. **Persistence: in-memory first, repository abstraction for Postgres compatibility**
   - **Decision:** Implement in-memory repositories now and define repository interfaces as the only persistence dependency for services.
   - **Rationale:** Delivers fastest MVP while preserving replacement path for a Postgres adapter.
   - **Alternatives considered:** SQLite-first. Rejected for this change because the user prioritized speed and can defer durable storage.

5. **Domain identifier strategy compatible with multiple stores**
   - **Decision:** Generate string IDs in application code instead of relying on database auto-increment semantics.
   - **Rationale:** Prevents coupling with in-memory or SQLite-specific behavior and eases future Postgres adoption.
   - **Alternatives considered:** Numeric IDs generated by persistence layer. Rejected due to migration complexity and inconsistent semantics.

6. **MVP API scope restricted to study registration**
   - **Decision:** Expose registration-focused API endpoints and corresponding frontend forms/views only for this change.
   - **Rationale:** Protects timeline and avoids scope creep.
   - **Alternatives considered:** Include all outlined product features in v1. Rejected as infeasible within one-day delivery.

## Risks / Trade-offs

- **[Risk] In-memory persistence loses data on restart** -> **Mitigation:** Seed deterministic mock data at startup and document expected behavior.
- **[Risk] Future Postgres migration can still require contract changes** -> **Mitigation:** Keep repository interfaces narrow and explicit, and avoid persistence-specific behavior in handlers/services.
- **[Risk] MVP can be mistaken for production-ready** -> **Mitigation:** Clearly document non-goals and temporary constraints in README and change artifacts.
- **[Risk] Time-box pressure may reduce test depth** -> **Mitigation:** Prioritize service-level tests for registration validation and repository contract behavior.
